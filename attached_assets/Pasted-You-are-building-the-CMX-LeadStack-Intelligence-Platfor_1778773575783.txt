CODEBASE MAP (read this first)
==========================================================================

Monorepo structure:
  artifacts/api-server/src/       — Express API server
    routes/                       — All HTTP route handlers
    services/                     — Business logic and ingestors
    cron/index.ts                 — Scheduled jobs (node-cron)
  artifacts/web/src/              — React/Vite frontend (Wouter routing)
    pages/                        — One file per route
    components/                   — Shared UI components
  lib/db/src/schema/              — Drizzle ORM schema (5 files)
  scripts/src/                    — One-off CLI scripts (NPI import, etc.)

Key schema tables:
  facilities              — master facility list (keyed by NPI)
  account_facilities      — JOIN table: links facilities to tenant accounts
  purchase_signals        — all buy-intent signals across all sources
  con_filings             — CON application records from state portals
  equipment_records       — per-facility equipment with urgency tiers
  financial_documents     — 990s, HCRIS reports, bond documents
  facility_contacts       — contacts per facility with confidence scores
  con_alert_subscriptions — per-user CON alert subscriptions
  con_alert_notifications — delivered alert records

Already-implemented ingestors (in services/):
  nppesIngestor, fda510kIngestor, fdaRecallsIngestor, fdaMaudeIngestor,
  fdaClassificationIngestor, propublica990Ingestor, cmsDataIngestor,
  secEdgarIngestor, usaSpendingIngestor, clinicalTrialsIngestor,
  conFilingsIngestor, conAlertNotifier, signalScorer

Signal scoring uses WEIGHTS map in signalScorer.ts — tenant-agnostic 
global score + per-tenant engagement score on account_facilities.

CRM integration: GHL (GoHighLevel), HubSpot, Salesforce via OAuth. 
Credentials stored AES-256-GCM encrypted on sub_accounts.crm_credentials.

Frontend routes: /dashboard, /facilities, /signals, /con-filings, 
/contacts, /campaigns, /sequences, /drafts, /batches, /reports, 
/admin (tabs: Accounts, Sub-Account CRM Credentials, CON Review Queue, 
Key Rotation), /settings

--- 1B. EMMA Municipal Bond Ingestor ---
File: artifacts/api-server/src/services/emmaIngestor.ts

EMMA does not have a formal REST API, but the MSRB provides an EMMA API 
at https://emma.msrb.org/api/. Use the search endpoint:
  GET https://emma.msrb.org/api/SecSearch.aspx?type=O&category=bond&
      keywords=hospital+imaging+equipment&pageSize=25

Parse the JSON response for municipal bond issuances where:
  - issuerType includes "Hospital" or "Health"  
  - principalAmount > 5000000 (bonds over $5M)

For each qualifying bond:
  - Try to match to a facility by issuerName vs facilities.name (ILIKE)
  - Insert into financial_documents table with doc_type = 'municipal_bond'
  - Insert a purchase_signal with signal_type = 'bond_issued'
  - Store the EMMA detail URL as sourceUrl

Also query: https://emma.msrb.org/api/IssuerSearch.aspx for healthcare issuers 
in target states: IL, MI, NY, VA, CT, MD, KY, MS, AL, GA, MA

--- 1C. HCRIS Cost Reports Ingestor ---
File: artifacts/api-server/src/services/hcrisIngestor.ts

Source: CMS HCRIS API at https://data.cms.gov/provider-compliance/cost-report/hospital-provider-cost-report/api
(JSON REST, no key required)

Fetch the latest cost report records. For each record:
  - Match facility by PRVDR_NUM (provider number) against facilities.cmsId
    OR by PRVDR_CTRL_TYPE_CD and facility name similarity
  - Upsert into financial_documents with doc_type = 'hcris_cost_report'
  - Calculate depreciation ratio: equipment depreciation / total assets
  - If depreciation ratio > 0.40, insert purchase_signal with 
    signal_type = 'hcris_depreciation_spike'
  - Set confidence proportional to the depreciation ratio (0.4 = 60%, 0.7 = 90%)

Key fields to capture and store in parsedJson:
  PRVDR_NUM, PROVIDER_NAME, COST_RPT_STATUS_CD, FY_BGN_DT, FY_END_DT,
  NET_PATIENT_REV_AMT, TOT_OPERATING_EXPENSE_AMT, TOT_ASSETS_EOY_AMT,
  CAPITAL_EXPENDITURES (compute from available depreciation fields)

==========================================================================
IMPORTANT CONSTRAINTS
==========================================================================

1. All ingestors MUST be idempotent (re-runnable without duplicates). 
   Use ON CONFLICT DO NOTHING or ON CONFLICT DO UPDATE with 
   unique indexes.

2. All ingestors MUST handle network failures gracefully — wrap each 
   HTTP call in try/catch, log the error, and continue to the next record.
   Never let one bad API response fail the entire ingestor run.

3. Rate limiting: add a 200-300ms delay between API calls within each 
   ingestor. Use a simple sleep helper: 
   const sleep = (ms: number) => new Promise(r => setTimeout(r, ms))

4. All new API endpoints must use the existing requireAccount middleware 
   (or requireAdmin for admin-only endpoints) — never skip auth.

5. All new frontend pages must use the existing QueryClient hooks pattern 
   (react-query via @tanstack/react-query) — don't use raw fetch() calls.

6. New schema columns require a Drizzle migration — never modify the DB 
   directly. Run db:generate then db:migrate.

7. TypeScript strict mode is on — no `any` types, no ts-ignore comments.

8. The existing test for account_facilities isolation (one tenant's data
